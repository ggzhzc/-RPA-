# 影刀翻页批量发货：三种可行方案全记录

## 一、案例目标与背景
- **页面**：订单管理列表页，包含分页和多条待发货订单。
- **操作**：对每一页的每条待发货订单，先点击“确认发货”按钮，再点击弹窗中的“确定”按钮。
- **最终效果**：所有页、所有订单均被处理，无遗漏，无死循环。

## 二、核心概念速览
- **循环相似元素（web）**：一次性抓取当前页面所有相似元素形成静态列表，依次循环处理，列表不随操作后的页面变化而改变。
- **获取相似元素列表（web）**：只抓取列表，不自动循环，可配合其他循环指令使用。
- **等待**：翻页或弹窗操作后必须等待页面渲染完毕，否则元素抓取不完整。
- **动态刷新列表**：每完成一页操作或一次操作后重新获取元素列表，以反映最新页面状态。

---

## 三、方案一：双层“循环相似元素”嵌套（最推荐）

### 适用场景
页面有**明确的页码数字按钮**，且每个订单的发货按钮特征统一（如文本均为“确认发货”）。

### 流程结构
```
1. 获取已打开的网页对象 → 影刀商城
2. 循环相似元素(web)  → 页码相似元素 → 循环页码
   │
   ├─ 点击元素(web)         → 点击 循环页码
   ├─ 等待 1 秒              // 等待新页面订单加载
   │
   ├─ 循环相似元素(web)      → 确认发货 → 循环确认发货
   │    ├─ 点击元素(web)      → 点击 循环确认发货
   │    ├─ 点击元素(web)      → 点击 确定
   │    └─ [可选] 等待 0.5 秒
   │
   └─ 循环结束标记           // 内层结束
3. 循环结束标记              // 外层结束
```

### 原理说明
- **外层页码循环**：按页码顺序依次翻页，最后一页的页码处理完后循环自动结束，无需额外条件。
- **内层发货循环**：在当前页一次性抓取所有“确认发货”按钮，即使某个按钮被点击后状态变化，静态列表仍会继续处理剩余按钮。
- **等待**：翻页后必须加等待，否则内层抓取时页面尚未完成渲染，导致元素列表不完整。

### 优点
- 逻辑清晰，与人工操作思维一致。
- 无动态条件判断，稳定性极高，无死循环风险。
- 易于维护和调试。

### 配置关键点
1. 页码捕获：必须捕获所有具体页码数字（如1、2、3），**不能只捕获“下一页”箭头**。
2. 发货按钮捕获：使用文本匹配“确认发货”，不依赖位置或样式。
3. 等待时长：翻页后≥1秒；若弹窗关闭有延迟，可在“确定”后加0.5秒。

## 四、方案二：While 条件 + 动态刷新列表

### 适用场景
页码范围不固定、没有可捕获的页码数字（如滚动加载、只有“下一页”按钮），但**需要能判断当前页是否存在待发货订单**。

### 流程结构
```
1. 获取已打开的网页对象 → 影刀商城
2. 获取相似元素列表(web) → 确认发货 → 保存到 列表A    // 初次抓取
3. While 条件 (列表A 的长度 > 0 或 列表A 包含“确认发货”)
   │
   ├─ 循环相似元素(web) → 确认发货 → 循环确认发货   // 处理当前页所有发货
   │    ├─ 点击元素(web)  → 点击 循环确认发货
   │    ├─ 点击元素(web)  → 点击 确定
   │    └─ 等待 0.5 秒
   │
   ├─ 点击元素(web)        → 点击 下一页按钮
   ├─ 等待 1 秒
   │
   └─ 获取相似元素列表(web) → 确认发货 → 覆盖保存到 列表A   // 刷新列表
4. 循环结束标记
```

### 原理说明
- While 条件根据“当前页是否有发货按钮”决定是否继续循环。
- 内层仍使用“循环相似元素”一次性处理当前页所有按钮。
- **核心**：每点完一页并翻页后，**必须用“获取相似元素列表”重新抓取并覆盖列表A**，否则条件判断基于旧数据，会导致死循环（一直有发货按钮）或提前退出。

### 优点
- 不依赖页码元素，适应性强（如列表页使用无限滚动、动态加载）。
- 只要翻页按钮存在即可运行。

### 缺点
- 需要仔细配置动态刷新，若忘记刷新列表则会造成死循环。
- 依赖“下一页”按钮，若最后一页该按钮仍存在但失效，可能需要额外条件退出。

### 配置关键点
1. While 条件：推荐使用 **列表A 的长度 > 0**（通过长度变量或直接比较），避免文本包含判断的不稳定性。
2. 刷新列表位置：**必须在点击下一页并等待后立即执行**，且要覆盖原列表变量。
3. 若最后一页“下一页”按钮消失或不可点击，需用异常处理或其它方式终止循环。

## 五、方案三：页码循环 + ForEach 列表

### 适用场景
有明确页码数字，且希望内层使用 ForEach 对当前页发货按钮进行遍历。

### 流程结构
```
1. 获取已打开的网页对象 → 影刀商城
2. 循环相似元素(web) → 页码相似元素 → 循环页码
   │
   ├─ 点击元素(web)       → 点击 循环页码
   ├─ 等待 1 秒
   │
   ├─ 获取相似元素列表(web) → 确认发货 → 保存到 发货列表
   ├─ ForEach 列表循环 → 遍历 发货列表 → 当前元素 变量
   │    ├─ 点击元素(web)   → 点击 当前元素
   │    ├─ 点击元素(web)   → 点击 确定
   │    └─ 等待 0.5 秒
   │
   └─ 循环结束标记        // ForEach 结束
3. 循环结束标记           // 外层页码循环结束
```

### 原理说明
- 外层页码循环控制翻页。
- 内层使用 **ForEach 列表循环**，需提前用“获取相似元素列表”准备好当前页的发货按钮列表。
- ForEach 同样会遍历一个静态列表，因此即使点击后按钮状态改变也不会中断循环。

### 优点
- 逻数直观，页码固定，循环次数明确。
- ForEach 循环变量可直接用于点击，适合习惯使用列表变量的场景。

### 缺点
- 需要额外配合“获取相似元素列表”指令，步骤比方案一稍多。
- 若页面相同，方案一直接用“循环相似元素”更简洁。

### 配置关键点
1. 页码捕获同方案一，需具体页码数字。
2. “获取相似元素列表”的特征必须稳定，以免漏抓。
3. ForEach 内点击使用循环中的“当前元素”变量，避免使用固定的捕获元素（因为列表可能变化）。

---

## 六、三方案对比速览

| 方案 | 外层控制 | 内层控制 | 适用场景 | 复杂度 | 稳定性 |
|------|----------|----------|----------|--------|--------|
| 方案一 | 循环相似元素(页码) | 循环相似元素(确认发货) | 页码明确，按钮特征统一 | ★☆☆☆☆ | ★★★★★ |
| 方案二 | While + 列表长度判断 | 循环相似元素(确认发货) | 页码无规律，只有“下一页” | ★★★☆☆ | ★★★★☆（需注意刷新） |
| 方案三 | 循环相似元素(页码) | ForEach + 获取列表 | 页码明确，习惯使用列表变量 | ★★☆☆☆ | ★★★★★ |

## 七、总结
- **首选方案一**：最简洁、最稳定，直接嵌套两层“循环相似元素”即可。
- **方案二为特殊环境备选**：当没有页码数字时使用，务必注意动态刷新列表。
- **方案三提供另一种写法**：适合偏好变量控制或需要更细粒度操作的场景。
